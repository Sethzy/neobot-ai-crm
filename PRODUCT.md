# Product

## Register

product

## Users

NeoBot is used by solo practitioners in advisory sales: real estate agents, insurance advisors, financial planners, and similar client-facing operators. They use the app between meetings, on mobile while moving between client work, and on desktop for focused review, CRM cleanup, approvals, and automation setup.

## Product Purpose

NeoBot is an autopilot for advisory sales work. The product should make completed work visible and controllable: CRM updates, follow-ups, meeting prep, inbound handling, automations, approvals, and memory. Success means the practitioner can trust the system, review the agent's work quickly, approve external-facing actions confidently, and recover context without hunting.

## Brand Personality

Competent, calm, operational. The interface should feel like a trusted workbench for high-stakes client work, not a novelty AI demo or a decorative dashboard.

## Anti-references

Avoid generic AI SaaS aesthetics: oversized empty hero cards inside the app, decorative gradients, glassmorphism, noisy metric-card grids, vague assistant chrome, and one-note purple/blue palettes. Avoid desktop-only CRM interfaces that collapse into horizontal scroll on mobile without a task-specific mobile path.

## Design Principles

1. Keep the work primary. Chat, CRM records, automations, approvals, and memory should stay close to the user's current task.
2. Mobile is not a reduced desktop. Preserve core workflows on phones with touch-first navigation, progressive disclosure, and full-screen detail/edit patterns.
3. Use earned familiarity. Standard product patterns, predictable navigation, and consistent controls are preferable to invented affordances.
4. Make trust inspectable. Agent output, record edits, approvals, and automation state need clear provenance, status, and recovery paths.
5. Keep density calm. Desktop can be information-rich, but spacing, hierarchy, and token usage should make repeated scanning feel controlled.

## Accessibility & Inclusion

Target WCAG AA. Interactive controls should have clear labels, visible focus states, and 44px touch targets on touch layouts. Color should use the existing Flexoki semantic token system rather than raw palette classes so light and dark themes remain legible.
